# 第1章 全体方針

## 1.1 目的

| 目的 | 説明 | 主担当 |
|---|---|---|
| 普及 | 見学希望者との接点を作り、見学→入会の転換率を高める | 普及委員会 |
| 定着 | 会員のMS出席と実践の習慣化を支え、休眠・退会を防ぐ | 朝礼・研修委員会 |
| 省力化 | 役員の連絡業務(電話・個別メール・出欠集計)を自動化する | 事務局 |

## 1.2 役割分担の基本思想

- **Webサイト = 機能の本体**(ストック)
  検索からの入口、情報のアーカイブ、そして将来的には出欠・実践記録・運営分析の機能を持つ「データベースと画面」。
- **公式LINE = 入口と通知エンジン**(フロー)
  タイミングよく届ける・行動を促す・習慣化を支える。入力や閲覧の実体はサイト(LIFF)へ誘導する。
- **両者はLINEログイン(LIFF)で統合**
  「LINEの友だちID = 会員ID」として紐づけ、誰が・いつ・何をしたかを同一IDで記録する。

## 1.3 システム全体構成

```
[会員・見学者・未入会者]
        │
        ▼
[LINE公式アカウント] ── 通知・配信・1to1チャット
        │
   ┌────┼────────┐
   ▼    ▼        ▼
[LINEログイン] [LIFF] [Messaging API]
 本人認証    LINE内表示  自動通知送信
   │    │        │
   ▼    ▼        ▼
[公開サイト] [会員ポータル] [運営ダッシュボード]
 集客・SEO   出欠・実践記録   出席率・休眠検知
   └────┴────────┘
              │
              ▼
        [データ基盤]
   会員・出席・実践・申込の記録
              │
              └──(条件成立で)──▶ 自動通知としてLINEへ戻る
```

## 1.4 展開計画(対象者を段階的に広げる)

対象者を **役員 → 会員 → 未会員** の順に拡大する。機能を作ってから一斉に配るのではなく、
狭い対象で運用を固め、移行条件を満たしてから次の層へ開放する。

| 段階 | 対象 | 期間目安 | Webサイト | LINE |
|---|---|---|---|---|
| Stage 1 試験運用 | **役員のみ**(10〜20名) | 〜2か月 | 限定URLで役員レビュー(実情報への差し替え・誤字確認)。出欠フォームを役員で試験 | 役員のみ友だち追加。週次配信・出欠回答・設営当番通知を本番テンプレ通りに回し、文面・頻度・操作性を検証 |
| Stage 2 会員展開 | 全会員 | 3〜5か月 | サイト本公開。講話スケジュール・活動報告の定期更新を開始。出欠フォーム本番運用 | MS・会報・例会で登録を案内。週次配信・水曜催促・前日リマインド開始。シナリオC(新入会員)稼働 |
| Stage 3 未会員展開 | 見学者・未会員 | 6か月〜 | SEO対策・Googleビジネスプロフィール登録。見学申込フォーム本格運用 | 経路別QRを配布(MS受付・チラシ・紹介カード)。シナリオA・B稼働。普及ファネル計測開始 |
| Stage 4 高度化 | 全層 | 7か月〜 | 会員ポータル(LIFF)・QRチェックイン・実践記録・運営ダッシュボード | 休眠アラート等のトリガー配信、リッチメニュー出し分け |

### Stage 1(役員試験運用)のチェックリスト
- [ ] 役員全員が友だち追加し、出欠回答を最低2回体験する
- [ ] 週次配信を2〜4週回し、開封率・回答率を確認して文面を修正する
- [ ] サイトの実情報差し替え完了(会名・会場・電話・LINE URL・QR画像・地図)
- [ ] 個人情報の取り扱い(§5.2)と閲覧権限を役員会で承認する
- [ ] 配信・更新の担当者(§5.1)を決め、操作手順書を1往復させる

### 次の段階へ進む条件(ゲート)
| 移行 | 条件の目安 |
|---|---|
| Stage 1 → 2 | 役員の出欠回答率90%以上/配信作業が週30分以内で回る/役員会承認 |
| Stage 2 → 3 | 会員のLINE登録率60%以上/出欠回答率70%以上/苦情・トラブルゼロ |
| Stage 3 → 4 | 普及ファネルの計測が3か月分蓄積/ポータルに載せるデータ基盤(§2.5)が整備済み |

### 2.2.3 更新運用(誰でもできる設計)

更新データは `js/data.js` の `SCHEDULE` / `REPORTS` 配列に分離してある。広報委員はこの1ファイルだけを書き換えればよい(構造=index.html・体裁=css/style.css・描画=js/render.js には触れない)。

```js
// js/data.js
const SCHEDULE = [
  { date:"6/18", day:"木", theme:"演題", speaker:"会社名 役職 氏名 氏", tag:"経営体験" },
];
const REPORTS = [
  { date:"2026.6.5", title:"タイトル", summary:"1行要約", url:"" },  // url は省略可
];
```

- 更新担当:広報委員(月1回、月末に翌月分を入力)
- 差し替え必須箇所:会名/会場・住所/電話番号/LINEのURL(`https://lin.ee/XXXXXXX`)/QR画像/地図埋め込み/`index.html` 内の canonical・OGP の URL/JSON-LD の住所・電話
- 公開方法:任意の静的ホスティング(Netlify・Cloudflare Pages・レンタルサーバー等)に `official_line/` 一式(index.html・css・js)をそのまま配置

## 2.5 データ設計(Phase 2以降の基盤)

最初にこのテーブル構成を決めてから機能を積む。スプレッドシート運用でもDB運用でも同じ構造を使う。

### members(会員マスタ)
| 列 | 型 | 説明 |
|---|---|---|
| member_id | ID | 会員ID(主キー) |
| line_user_id | 文字列 | LINEのユーザーID(LIFF初回利用時に紐づけ) |
| name / company / role_in_kai | 文字列 | 氏名/社名/会内役職 |
| status | 列挙 | 未見学/見学者/会員/役職者/退会(§3.2の状態タグと同期) |
| joined_at / left_at | 日付 | 入会日/退会日 |
| source | 列挙 | 登録経路(サイト/MS受付/講演会/紹介/他) |

### schedule(講師スケジュール)
date / theme / speaker_name / speaker_title / tag / venue / note
→ サイト表示・週次配信・前日リマインドのすべてをこの1表から生成(一元管理)。

### attendance(出席)
member_id / date / declared(出席予定・欠席・遅参・未回答)/ checked_in(実出席フラグ)/ checkin_at

### practice(実践記録)
member_id / month / theme / declared_at / daily_log(日別フラグ)/ streak / reflection / share_ok

### events / event_entries(行事と申込)
event_id / title / date / capacity / fee — member_id / entry_at / attended

### follow_log(フォロー記録)
member_id / alert_date / assigned_to / contacted_at / result
→ 休眠アラートの「出しっぱなし」を防ぎ、対応完了まで追跡する。

# 第3章 公式LINE設計

## 3.1 アカウント基本設計

| 項目 | 内容 |
|---|---|
| アカウント名 | 岡崎市葵倫理法人会 |
| あいさつメッセージ | 登録御礼+MS紹介1行+サイトリンク+「見学希望」と送れば案内が届く旨 |
| 応答 | 営業時間内:事務担当が1to1対応/時間外:自動応答で受付の旨を返す |
| プラン | 会員150名規模で月3,000〜5,000通想定。従量プランを確認し、対象絞り込みで通数最適化 |

## 3.2 友だち分類とタグ体系

**状態タグ(排他・必ず1つ)**
`S1_未見学` → `S2_見学者` → `S3_会員` → `S4_役職者` / `S5_退会者`(配信停止・削除しない)

**経路タグ(登録時自動・分析用)**
`R_サイト` / `R_MS受付` / `R_講演会` / `R_紹介カード` / `R_その他` — 経路別の友だち追加QRを発行して計測。

**行動タグ(随時・複数可)**
`A_出席予定` `A_実践宣言中` `A_研修関心` `A_休眠予備` `A_朝食会`

## 3.3 リッチメニュー

**会員向け(3×2)**:今週のセミナー/出欠連絡/会場アクセス/実践記録(当面は活動報告)/行事・研修/お問い合わせ
**未見学・見学者向け(2×2)**:見学を申し込む/倫理法人会とは/今月の講師/よくある質問

> 出し分けはLステップ等の拡張ツール、または Messaging API の richmenu API(ユーザー単位割当)で実装。導入前は会員向けを共通メニューとして使用。

## 3.5 ステップ配信シナリオ

|---|---|

## 3.6 実践サポートの月次ループ

1. 月初:実践テーマ宣言募集(挨拶/早起き/整理整頓/自由記述)→ `A_実践宣言中` 付与
2. 毎週月曜:宣言者のみに継続確認(継続中/中断 のワンタップ)
3. 月末:振り返りフォーム(継続日数・気づき・紹介可否)
4. 翌月初:紹介可の事例を配信&MS会長挨拶で承認(表彰)
5. ポータル実装後:毎日のワンタップ記録+連続日数表示に発展

**設計意図:宣言→リマインド→承認のループが回ると、欠席した週も会とのつながりが切れない。**

# 第4章 サイト×LINE 連携仕様

## 4.1 ID統合の手順

1. 会員がリッチメニューからポータル(LIFF)を開く
2. LIFFがLINEユーザーIDを自動取得(ログイン操作不要)
3. 初回のみ:名簿から自分の名前を選び確認 → members.line_user_id に保存
4. 以後、出欠・実践・申込の全操作が会員IDに自動で紐づく

## 4.2 トリガー配信一覧(データ起点の自動通知)

| トリガー条件 | アクション |
|---|---|
| 見学フォーム送信 | 申込者へ自動返信+普及委員へ通知+`S1`タグ確認 |
| 受付チェックイン | `S2`付与(初見学時)/出席実績記録/当日17時の御礼配信を予約 |
| 入会登録 | `S3`付与/シナリオC起動/オンボーディング用ページ案内 |
| 出欠未回答(水曜) | 未回答者のみ催促配信 |
| 行事申込 | 受付完了返信+前日リマインド自動予約 |
| 実践記録の中断検知 | 「昨日の記録がまだです」(宣言者のみ・任意でON/OFF可) |

## 4.3 休眠検知ロジック

| 段階 | 条件 | アクション |
|---|---|---|
| 注意 | 連続4週 MS欠席 | ダッシュボードの注意リストに掲載(本人配信なし) |
| アラート | 連続6週欠席+実践記録なし+配信未開封 | `A_休眠予備` 付与 → **担当役員へ「◯◯さんフォローのタイミングです」と通知** |
| 対応 | 役員が電話・訪問・MSでの声かけ | follow_log に対応記録。翌週内対応率100%を目標 |

原則:**機械的な「来てください」を本人に送らない。システムは人が動くきっかけを作る。**

# 第5章 運用体制・セキュリティ

## 5.1 役割分担

| 役割 | 担当 | 工数目安 |
|---|---|---|
| 配信責任者(最終確認) | 広報委員長 | 週10分 |
| 配信オペレーター | 広報委員 1〜2名 | 週20分 |
| サイト更新(講師・報告) | 広報委員 | 月30分 |
| 1to1チャット一次対応 | 事務担当 | 随時(翌営業日まで) |
| 見学フォロー(シナリオB個別連絡) | 普及委員 | 随時 |
| タグ・名簿・権限管理 | 事務担当 | 入退会時 |

## 5.2 個人情報・セキュリティ

- 出席履歴・実践記録・チャット内容はセンシティブな個人データとして扱う
- 閲覧権限:ダッシュボードは役職者のみ。役割ごとにアカウントを分け、共有パスワードを使わない
- 同意:入会時・ポータル初回利用時に「何を記録し、誰が見られるか」を明示して同意取得
- 期末の役員交代時に権限の棚卸しを必ず実施
- 退会者:配信停止(`S5`)とし、データは規程に沿って保管・削除

## 5.3 属人化の防止

- 本書+管理画面の操作スクリーンショット手順書を毎期更新
- 実装は標準的な構成(§2.6)を選び、凝った独自実装を避ける
- データ構造(§2.5)を文書化しておけば、ツールを乗り換えても資産が残る

## 5.4 その他の注意

- 上位組織(県・法人局)の方針・システムとの整合を事前確認
- 開封率が落ちた配信枠は減らす。「送れるから送る」をしない
- 『万人幸福の栞』『職場の教養』等の本文をサイト・配信に転載しない

# 第6章 KPIと振り返り

月次で広報委員会にて確認し、四半期で施策を見直す。
KPIは展開段階に応じて計測を開始する(Stage 1:開封率・出欠回答率を役員内で試験計測/Stage 2:会員系指標/Stage 3:普及系指標)。

| 指標 | 定義 | 目標の目安 |
|---|---|---|
| 友だち数 | 累計(ブロック除く) | 会員数 × 1.5以上 |
| 配信開封率 | 週次配信 | 60%以上 |
| 出欠回答率 | 月曜配信→木曜まで | 会員の70%以上 |
| MS平均出席率 | 実出席 ÷ 会員数 | 前年比+5pt |
| 見学転換率 | LINE登録→見学 | 30% |
| 入会転換率 | 見学→3か月以内入会 | 25% |
| 実践宣言率 | 会員のうち月次宣言者 | 40% |
| 休眠フォロー率 | アラート→翌週内接触 | 100% |
